# Aster 执行文档

## 1. 项目定位

Aster 是面向酒店内部使用的物资运营管理客户端，用来替代当前以 Excel 为主的易耗品进销存管理方式。

项目目标不是简单复刻 Excel，而是建立一套稳定的业务结构：

- 物品档案统一维护
- 入库、出库、盘点、调整全部形成单据和库存流水
- 出库按部门领用处理
- 库存余额由流水自动计算
- 月报、部门报表、库存预警从结构化数据生成
- 支持旧 Excel 数据迁移、校验和导出

技术栈：

- 桌面客户端：Tauri 2
- 前端界面：React
- 本地后端：Rust
- 本地数据库：SQLite
- 文件能力：Excel 导入、Excel 导出、本地备份、恢复
- 目标平台：Windows + macOS 双端客户端

## 2. 业务原则

1. 流水优先
   所有库存变化都必须产生库存流水，报表和余额不能依赖手工填写的汇总值。

2. 单据驱动
   入库、出库、盘点、调整都以单据为业务入口，一张单据可以包含多行物品。

3. 出库就是部门领用
   系统内统一使用出库流水表达部门领用，报表中按部门统计领用数量和金额。

4. 历史不可硬删
   已确认的业务数据原则上不直接删除。错误数据通过作废、冲正或调整处理。

5. 本地可靠
   所有关键数据保存在本地 SQLite，并提供备份、恢复、健康检查和主机故障切换能力。

6. Excel 兼容
   旧 Excel 是迁移来源，新系统仍要支持导出酒店日常能接受的 Excel 报表。

## 3. 功能模块总览

### 3.1 首页工作台

用途：让仓库或管理人员快速看到本月经营状态和异常。

功能：

- 本月入库金额
- 本月出库/领用金额
- 当前库存金额
- 当前库存品类数
- 低库存物品数量
- 负库存异常数量
- 最近操作流水
- 快捷入口：
  - 新建入库单
  - 新建出库/领用单
  - 导入 Excel
  - 导出月报
  - 创建盘点单

验收要求：

- 首页数据必须来自数据库统计，不写死。
- 最近操作至少展示日期、业务类型、物品、数量、部门或供应商。

### 3.2 物品档案

用途：维护所有可入库、出库、统计的物品基础信息。

字段：

- 物品 ID
- 物品编码
- 物品名称
- 分类
- 规格
- 单位
- 参考进价，仅用于入库录入预填和无历史数据兜底
- 默认供应商
- 库存预警线
- 是否启用
- 备注
- 创建时间
- 更新时间

功能：

- 新增物品
- 编辑物品
- 启用/停用
- 搜索物品
- 按分类筛选
- 批量导入
- 重复物品提示
- 停用品保留历史记录，但不可在新单据中默认选择

验收要求：

- 支持 1000+ 物品不卡顿。
- 物品编码必须唯一。
- 名称、规格、单位是物品档案核心字段；参考进价只用于录入预填，真实库存成本必须来自入库批次。

### 3.3 分类与单位管理

用途：保证物品档案规范，减少录入混乱。

功能：

- 物品大类维护
- 物品小类维护
- 单位维护
- 分类排序
- 启用/停用分类

验收要求：

- 分类用于筛选和报表统计。
- 单位用于物品档案和单据展示。

### 3.4 部门管理

用途：维护领用部门，并作为部门报表的统计口径。

字段：

- 部门 ID
- 部门编码
- 部门名称
- 负责人，可选
- 是否启用
- 排序
- 备注

功能：

- 新增部门
- 编辑部门
- 启用/停用
- 排序
- 历史部门名称保留在历史记录中

默认部门可从旧 Excel 配置迁移：

- 行政办
- 餐饮
- 温泉+前台
- 客房
- 工程
- 安保
- 妇女之家
- 调物品

### 3.5 供应商管理

用途：记录采购来源，并支持采购记录查询和采购分析。

字段：

- 供应商 ID
- 供应商名称
- 联系人
- 电话
- 地址
- 备注
- 是否启用

功能：

- 新增供应商
- 编辑供应商
- 查看供应商采购记录
- 设置物品默认供应商

验收要求：

- 作为入库单可选字段。
- 可查看供应商采购记录。
- 暂不做应付账款和财务结算。

### 3.6 入库管理

用途：记录采购入库、退回入库、盘盈入库等增加库存的业务。

单据字段：

- 入库单号
- 入库日期
- 入库类型
- 供应商
- 经办人
- 备注
- 状态：草稿、已确认、已作废

明细字段：

- 物品
- 数量
- 单价
- 金额
- 采购批次号，系统生成
- 批次剩余数量
- 批次剩余金额
- 备注

功能：

- 新建入库单
- 一张单据多行物品
- 保存草稿
- 确认入库
- 作废入库单
- 查看入库明细
- 查看入库批次和批次剩余库存
- 按月份、供应商、物品筛选

验收要求：

- 确认后生成库存流水。
- 每一条确认后的入库明细都必须生成独立采购批次。同一个物品不同入库批次可以有不同进货价，系统必须保留原始批次价。
- 金额默认等于数量乘本次进价，可允许手工修正；确认后本次金额会折算为该入库批次实际采购价。
- 批次金额必须等于该批次当前剩余数量对应的成本余额，不能只用物品默认价或全局平均价替代。
- 已确认单据不能直接编辑核心数量，需走冲正或作废。

### 3.7 出库/部门领用

用途：记录部门领用物品并扣减库存。

单据字段：

- 出库单号
- 出库日期
- 领用部门
- 经办人
- 用途
- 备注
- 状态：草稿、已确认、已作废

明细字段：

- 物品
- 数量
- 成本单价
- 成本金额
- 扣减批次，系统按成本策略自动分摊
- 备注

功能：

- 新建出库单
- 一张单据多行物品
- 选择领用部门
- 保存草稿
- 确认出库
- 作废出库单
- 查看部门领用明细
- 按月份、部门、物品筛选

验收要求：

- 出库就是部门领用。
- 确认后生成库存流水。
- 出库成本必须从现有入库批次扣减，默认按先进先出 FIFO；同一出库明细可拆分消耗多个采购批次。
- 库存流水必须记录本次实际消耗的批次、批次采购价、消耗数量和成本金额，部门领用金额以实际批次成本为准。
- 默认不允许负库存，是否允许负库存由系统设置控制。
- 部门领用明细必须是实写记录或数据库查询结果，不依赖 Excel 动态公式。

### 3.8 库存台账

用途：查看每个物品的当前库存和金额。

字段：

- 物品编码
- 物品名称
- 分类
- 规格
- 单位
- 当前库存数量
- 当前库存金额
- 成本均价，按批次余额汇总计算
- 最近入库价
- 可用批次数
- 库存预警线
- 库存状态

功能：

- 查看当前库存
- 查看物品批次库存明细，包括入库日期、供应商、原始数量、剩余数量、批次采购价、剩余金额
- 按分类筛选
- 搜索物品
- 查看某个物品的库存流水
- 低库存提示
- 负库存提示

验收要求：

- 库存余额由库存流水汇总计算并缓存。
- 库存金额以入库批次剩余金额汇总为准；成本均价只是展示字段，不能作为唯一成本来源。
- 任何出库、盘亏、损耗和作废冲正都必须同步更新批次剩余数量、批次剩余金额、库存余额和库存流水。

### 3.9 库存流水

用途：记录所有影响库存的业务痕迹。

流水类型：

- 期初
- 入库
- 出库
- 盘盈
- 盘亏
- 调整
- 冲正

字段：

- 流水 ID
- 发生日期
- 物品
- 方向：入、出
- 数量
- 单价
- 金额
- 批次 ID，可为空；入库流水关联新批次，出库流水关联被消耗批次
- 批次采购价
- 关联单据类型
- 关联单据 ID
- 部门
- 供应商
- 操作人
- 备注
- 创建时间

验收要求：

- 流水不可直接修改。
- 报表、库存余额、部门领用明细都从流水生成。

### 3.10 盘点管理

用途：定期核对账面库存与实际库存差异。

功能：

- 创建盘点单
- 按全部物品、分类或自定义范围盘点
- 导出盘点表
- 录入实盘数量
- 计算盘盈盘亏
- 确认盘点并生成调整流水

验收要求：

- 支持创建盘点单。
- 支持导出盘点表。
- 支持录入实盘数量。
- 支持生成盘盈盘亏差异。
- 支持确认盘点并生成调整流水。

### 3.11 库存调整

用途：处理损耗、修正、盘点差异等非采购和非领用场景。

调整类型：

- 盘盈调整
- 盘亏调整
- 损耗调整
- 数据修正

要求：

- 必须填写原因。
- 必须生成库存流水。
- 涉及审批时按审批流程执行。

验收要求：

- 支持盘盈、盘亏、损耗、数据修正。
- 必须填写调整原因。
- 调整必须生成库存流水。

报表：

### 3.13 Excel 导入

用途：把整理好的 Excel 三表模板导入 Aster，生成真实物品档案、入库单、出库单、采购批次、库存流水和导入报告。

功能：

- 选择 Excel 文件
- 导出新版导入模板
- 识别 `物品档案`、`入库明细`、`出库明细` 三张工作表
- 识别物品编码、物品名称、分类、规格、单位、参考进价、参考售价和预警库存
- 识别入库业务时间、供应商、数量、进货单价和进货金额
- 识别出库业务时间、出库类型、部门、数量、销售单价、经办人和用途
- 导入前预览
- 异常校验
- 重复物品提示
- 导入报告
- 支持只导入物品档案
- 支持导入入库明细为真实采购批次
- 支持导入出库明细为内部领用或酒店客人销售

异常类型：

- 物品名称为空
- 单位为空
- 单价异常
- 数量异常
- 金额不平
- 公式错误
- 重复物品
- 缺少必填工作表
- 缺少必填表头

验收要求：

- 新版模板固定包含 `物品档案`、`入库明细`、`出库明细`。
- 导入前必须可预览，不直接写入正式库。
- 导入完成后生成迁移报告和导入前保护备份。
- 入库导入必须生成真实入库单和采购批次。
- 出库导入必须复用正常出库逻辑，按 FIFO 扣减批次；酒店客人销售必须记录销售收入、实际成本和毛利。

## 5. 多电脑一致性与冗灾方案

### 5.1 总体结论

Aster 不采用“多台电脑各自保存一份数据库再同步合并”的方式。

正式多电脑使用采用局域网主机模式：

- 一台电脑作为数据主机。
- 主机保存唯一正式 SQLite 数据库。
- 主机运行本地局域网服务。
- 其他电脑作为客户端连接主机。
- 所有写入、库存计算、报表统计都由主机完成。
- 客户端不直接写数据库。

这样可以保证所有电脑看到的是同一套实时数据，避免多份 SQLite 合并带来的冲突和库存不一致。

### 5.2 运行模式

Aster 支持三种运行模式：

1. 单机模式
   - 本机保存数据库。
   - 适合只有一台电脑使用。
   - 不开放局域网连接。

2. 主机模式
   - 本机保存正式数据库。
   - 启动局域网服务。
   - 允许其他客户端连接。
   - 负责自动备份、恢复、数据健康检查。

3. 客户端模式
   - 本机不保存正式业务数据库。
   - 通过主机 IP 和端口连接主机。
   - 所有业务操作通过主机 API 完成。
   - 可保留少量本机配置，例如最近连接地址和界面设置。

架构和界面都需要明确区分这三种模式，不能把数据库访问写死为“永远本机直连”。

### 5.3 一致性原则

1. 单一写入点
   只有主机可以写正式数据库。客户端提交入库、出库、盘点、调整时，必须通过主机服务。

2. 事务保护
   单据确认、库存流水生成、库存余额更新必须在主机同一个数据库事务中完成。

3. 并发控制
   多台客户端同时操作时，由主机按事务顺序处理，避免同时出库导致库存错误。

4. 版本校验
   关键编辑场景使用 updated_at 或 revision 字段做乐观锁，防止旧页面覆盖新数据。

5. 禁止共享 SQLite 文件
   不允许把 SQLite 数据库放到共享文件夹后让多台电脑直接打开写入。

### 5.4 主机服务能力

主机模式需要提供：

- 局域网 API 服务
- 连接测试接口
- 版本信息接口
- 数据库健康检查接口
- 当前连接客户端列表
- 业务 command/API
- 备份和恢复接口
- 导入导出接口

服务安全要求：

- 默认只监听局域网地址或本机配置允许的地址。
- 首次连接需要配对码或管理员确认。
- 客户端保存连接凭据。
- 危险操作，例如恢复备份、清空导入，必须只允许主机本机或管理员执行。

### 5.5 客户端连接能力

客户端模式需要提供：

- 主机 IP 和端口配置
- 自动发现主机
- 连接测试
- 断线提示
- 自动重连
- 主机不可用时禁止创建新业务单据
- 明确显示当前连接的是哪台主机

验收要求：

- 支持手动填写主机地址。
- 主机地址只填写 IP 或主机名，不包含 `http://`、端口或路径；端口默认 `17871`，有效范围为 `1024-65535`。
- 支持局域网自动发现主机。
- 配对码必须为主机显示的一次性 12 位数字，客户端名称和设备 ID 不能为空。
- 客户端断线后有明确提示，并按固定间隔自动检测主机可用性；主机恢复后无需重启客户端即可显示重新连接成功。

### 5.6 跨平台要求

Aster 客户端支持 Windows 和 macOS 双端。

要求：

- Windows 和 macOS 使用同一套 React 前端和 Rust 业务核心。
- 两端都支持单机模式、主机模式和客户端模式。
- Windows 主机可以被 macOS 客户端连接。
- macOS 主机可以被 Windows 客户端连接。
- 数据库 schema、备份包、导入导出文件在两端保持兼容。
- 本地文件路径必须通过 Tauri/Rust 的平台目录 API 获取，不硬编码系统路径。
- 备份包使用 zip 等跨平台格式。
- Excel 导入导出不依赖本机安装 Microsoft Excel 或 WPS。

平台差异处理：

- Windows 需要考虑防火墙放行主机端口提示。
- macOS 需要考虑局域网访问权限和应用签名/权限提示。
- Windows 默认数据目录、macOS 默认数据目录不同，但业务层不感知差异。
- 文件名避免使用平台不兼容字符。
- 所有路径在数据库中保存为规范化字符串，展示时按平台转换。

打包目标：

- Windows：安装包或可执行发布包。
- macOS：dmg 或 app 发布包。

### 5.7 冗灾目标

冗灾不是只做一个“导出数据库”按钮，而是要保证出现故障时能恢复到可接受的数据状态。

目标：

- 主机电脑损坏时，可以用备份在另一台电脑恢复。
- 数据库文件损坏时，可以回滚到最近可用备份。
- 误操作发生时，可以找到操作日志并通过作废、冲正或备份恢复处理。
- 备份文件本身要可校验、可恢复、可追踪。

恢复目标：

- RPO：最多丢失当天最近一次备份之后的数据。
- RTO：普通人员按文档操作，30 分钟内恢复到另一台电脑继续使用。

### 5.8 备份策略

主机模式执行自动备份，客户端不备份正式数据库。

备份类型：

1. 启动备份
   - 每天第一次启动主机时自动备份一次。

2. 定时备份
   - 主机运行时按配置定时备份，例如每 2 小时一次。

3. 操作前备份
   - 执行 Excel 批量导入、恢复备份、数据库迁移前自动备份。

4. 手动备份
   - 管理员可随时手动创建备份。

备份内容：

- SQLite 数据库
- 应用配置
- 导入报告
- 备份元数据

备份命名：

- aster-backup-YYYYMMDD-HHMMSS-短ID.zip

备份元数据：

- 应用版本
- 数据库 schema 版本
- 创建时间
- 主机名称
- 操作系统
- 数据库大小
- SHA256 校验值
- 备份原因：auto、manual、before_import、before_restore、before_migration

### 5.9 备份位置

建议至少保留两个位置：

1. 本机备份目录
   - 用于快速恢复。

2. 外部备份目录
   - 可以是移动硬盘、NAS、局域网共享目录或其他电脑共享目录。
   - 用于主机电脑损坏后的恢复。

验收要求：

- 支持配置本机备份目录。
- 支持配置第二备份目录。
- 第二备份目录不可用时，系统要在首页显示告警。

### 5.10 备份保留策略

默认策略：

- 最近 7 天：保留所有自动备份。
- 最近 8 到 30 天：每天保留最后一个备份。
- 最近 12 个月：每月保留最后一个备份。
- 手动备份默认不自动删除，除非用户主动清理。

备份清理必须保留日志。

### 5.11 数据库健康检查

主机需要定期执行：

- SQLite integrity_check
- 数据库文件可读写检查
- 最近备份是否成功
- 第二备份目录是否可写
- 库存余额和流水抽样核对
- 负库存异常统计

首页需要显示：

- 最近备份时间
- 最近备份结果
- 第二备份状态
- 数据库健康状态

### 5.12 恢复流程

恢复备份必须谨慎处理。

流程：

1. 用户选择备份文件。
2. 系统校验备份包结构。
3. 系统校验 SHA256。
4. 系统显示备份创建时间、应用版本、数据库版本。
5. 如果用户修改备份文件路径，必须重新校验，恢复按钮只对当前已校验路径生效。
6. 系统自动备份当前数据。
7. 系统停止业务写入。
8. 系统替换数据库。
9. 系统执行数据库健康检查。
10. 系统重启或刷新主机服务。
11. 系统写入恢复日志。

恢复失败时：

- 必须自动回滚到恢复前保护备份。
- 必须给出失败原因。

### 5.13 主机故障切换

如果主机电脑损坏，处理方式：

1. 在另一台电脑安装 Aster。
2. 选择主机模式。
3. 从外部备份目录选择最近备份。
4. 执行恢复。
5. 新主机显示新的 IP 和端口。
6. 其他客户端修改连接地址或重新发现主机。

验收要求：

- 支持手动切换主机。
- 支持自动发现新主机。
- 客户端可以修改连接地址并重新连接。

跨平台恢复要求：

- Windows 创建的备份可以在 macOS 恢复。
- macOS 创建的备份可以在 Windows 恢复。
- 恢复时重新计算本机默认路径，不直接沿用旧主机绝对路径。

### 5.14 明确不做的同步能力

Aster 当前全量范围不做：

- 多台电脑离线分别录入后自动合并。
- 多主机写入。
- 客户端本地缓存业务单据后断网提交。
- 云端实时同步。

这些能力需要事件同步、冲突解决和多主架构，和当前“局域网单主机一致性”路线冲突。

## 6. 核心业务流程

### 6.1 新增物品

1. 用户进入物品档案。
2. 点击新增物品。
3. 填写名称、分类、规格、单位、参考进价、预警线。
4. 系统校验编码唯一、必填字段完整。
5. 保存物品。
6. 物品可用于入库和出库单。

### 6.2 入库流程

1. 用户新建入库单。
2. 选择入库日期、供应商、入库类型。
3. 添加多行物品。
4. 填写数量、单价、备注。
5. 保存草稿或确认入库。
6. 确认后系统生成库存流水。
7. 系统按每一条入库明细生成采购批次，记录原始数量、剩余数量、采购单价、原始金额和剩余金额。
8. 系统更新库存余额和库存金额；库存金额来自批次剩余金额汇总。

### 6.3 出库/领用流程

1. 用户新建出库单。
2. 选择出库日期和领用部门。
3. 添加多行物品。
4. 填写数量、用途和备注。
5. 系统检查库存是否足够。
6. 保存草稿或确认出库。
7. 确认后系统按 FIFO 从可用采购批次中扣减数量；数量跨批次时自动拆分。
8. 系统生成出库流水和批次消耗明细，流水金额等于实际消耗批次成本。
9. 系统更新批次剩余数量、批次剩余金额、库存余额。
10. 部门领用报表可立即按实际批次成本查询。

### 6.4 月报流程

1. 用户进入报表中心。
2. 选择月份。
3. 系统按物品汇总期初、入库、出库、结存。
4. 系统按部门汇总领用数量和金额。
5. 用户查看或导出 Excel。

### 6.5 Excel 迁移流程

1. 用户选择旧 Excel 文件。
2. 系统解析文件结构。
3. 系统展示导入预览。
4. 系统列出异常和风险。
5. 用户选择导入范围。
6. 系统写入物品档案、期初和历史流水。
7. 系统生成导入报告。

## 8. 技术架构

### 8.1 前端 React

职责：

- 页面布局
- 表单交互
- 表格筛选
- 状态展示
- 调用 Tauri command
- 轻量本地状态管理

建议：

- 使用 React + TypeScript。
- 使用 Vite 作为前端构建工具。
- 表格组件要考虑 1000+ 物品和大量流水，必要时使用虚拟列表。
- 不在前端直接读写数据库。

### 8.2 Rust/Tauri 后端

职责：

- SQLite 读写
- 业务校验
- 库存计算
- Excel 导入
- Excel 导出
- 备份恢复
- 文件选择和本地路径处理
- 主机模式局域网服务
- 客户端模式远程调用主机服务

建议：

- Tauri commands 作为前后端边界。
- 核心业务逻辑放在 Rust service 层，不散落在 command 中。
- 数据库操作通过 repository 层隔离。
- 库存流水和余额更新必须在同一个事务中完成。
- 业务 service 要同时支持本机调用和主机 API 调用，避免单机模式与局域网模式写两套逻辑。

### 8.3 SQLite

职责：

- 存储业务数据
- 本地事务
- 支持备份和恢复

建议：

- 使用迁移文件管理 schema。
- 新增 `stock_batches` 表保存采购批次：
  - id
  - item_id
  - source_document_id
  - source_document_line_id
  - batch_no
  - inbound_date
  - supplier_id
  - supplier_name
  - original_quantity
  - remaining_quantity
  - unit_price
  - original_amount
  - remaining_amount
  - status
  - created_at
  - updated_at
- 新增 `stock_batch_movements` 表保存批次级消耗和冲正：
  - id
  - batch_id
  - stock_movement_id
  - document_id
  - document_line_id
  - direction
  - quantity
  - unit_price
  - amount
  - created_at
- `stock_movements` 增加批次关联字段，允许出库跨批次时生成多条流水或一条业务行对应多条批次流水。
- `stock_balances` 继续作为查询缓存，但数量和金额必须能由批次余额和流水重算校验。
- 对常用筛选字段建索引：
  - item code
  - item name
  - movement date
  - batch item id
  - batch remaining quantity
  - department id
  - document status
  - document type

### 8.4 局域网主机服务

职责：

- 只在主机模式启动。

### 8.6 跨平台打包与运行

## 11. 关键验收标准

   库存余额不是手工填的，而是由确认后的流水计算。

7. 多电脑一致
   主机模式下，所有客户端通过同一主机写入和查询，不出现多份数据库合并问题。

   主机损坏时，可以用第二备份目录中的备份在另一台电脑恢复为新主机。

- 客户端连接配置由服务层统一校验：地址只允许 IP/主机名，端口必须为 `1024-65535`，配对码必须为 12 位数字，客户端名称和设备 ID 不能为空。

- 主机配对成功后会把客户端名称、设备 ID、IP、版本、状态、最近连接时间和连接凭据哈希写入主机 SQLite 的 `client_connections`，设置页客户端列表从持久记录读取；同一设备 ID 重新配对会更新同一条客户端记录，不生成重复设备；客户端有效请求会刷新最近连接时间；主机服务重启后可用持久化凭据哈希恢复已配对客户端认证，不在连接列表暴露明文 token。
- 客户端未配对或主机连接未恢复时，库存、盘点、报表、预算、审批和导入等业务入口会暂停操作，只保留系统设置用于测试连接、修改主机地址和重新配对。
- 退出登录、切换运行模式、客户端未配对或主机连接未恢复时，前端会清空旧业务列表、报表、导入预览和管理员数据，避免继续展示上一登录用户或上一主机的缓存结果。
- 客户端刷新状态时若主机不可达，会回退到本机 shell 状态并显示连接异常，不继续拉取主机业务数据，避免断线状态下反复触发远程业务请求。
- 客户端模式未登录时只显示本机连接配置和主机健康探测结果，不读取主机业务状态、库存、报表、设置或审计数据；未配对客户端使用本机管理员兜底登录后也只加载本机主机/客户端配置，不加载主机业务数据；配对并使用主机用户登录后再按角色权限访问主机 SQLite。
- 客户端模式下首页统计、最近操作流水、主机数据库健康状态和业务数据列表均通过主机 API 读取主机 SQLite，本机只保留连接配置和客户端凭据。
- 客户端模式下，库存单据、库存调整、作废、库存台账、库存流水、基础资料、盘点和报表查询均通过主机 API 访问主机 SQLite。
- 主机业务 API 校验客户端连接凭据和远程当前用户权限，并刷新客户端最后访问时间；系统状态至少要求报表查看权限，系统设置、客户端连接清单、用户、角色、备份记录、预算和审批管理要求管理员权限，防止客户端绕过前端按钮直接调用接口。
- 主机 API 业务错误统一返回 JSON 错误原因，客户端可展示明确的权限、校验或恢复失败信息。
- 主机 HTTP 请求按 `Content-Length` 读取完整请求体，支持多行单据和多行盘点提交。
