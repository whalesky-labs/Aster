# Aster 执行文档

## 1. 项目定位

Aster 是面向酒店内部使用的物资运营管理客户端，用来替代当前以 Excel 为主的易耗品进销存管理方式。

项目目标不是简单复刻 Excel，而是建立一套稳定的业务结构：

- 物品档案统一维护
- 入库、出库、盘点、调整全部形成单据和库存流水
- 出库按部门领用处理
- 库存余额由流水自动计算
- 月报、部门报表、库存预警从结构化数据生成
- 支持旧 Excel 数据迁移、校验和导出

技术栈：

- 桌面客户端：Tauri 2
- 前端界面：React
- 本地后端：Rust
- 本地数据库：SQLite
- 文件能力：Excel 导入、Excel 导出、本地备份、恢复

## 2. 业务原则

1. 流水优先
   所有库存变化都必须产生库存流水，报表和余额不能依赖手工填写的汇总值。

2. 单据驱动
   入库、出库、盘点、调整都以单据为业务入口，一张单据可以包含多行物品。

3. 出库就是部门领用
   系统内统一使用出库流水表达部门领用，报表中按部门统计领用数量和金额。

4. 历史不可硬删
   已确认的业务数据原则上不直接删除。错误数据通过作废、冲正或调整处理。

5. 本地可靠
   第一阶段优先保证单机客户端稳定可用，所有关键数据保存在本地 SQLite，并提供备份恢复。

6. Excel 兼容
   旧 Excel 是迁移来源，新系统仍要支持导出酒店日常能接受的 Excel 报表。

## 3. 功能模块总览

### 3.1 首页工作台

用途：让仓库或管理人员快速看到本月经营状态和异常。

功能：

- 本月入库金额
- 本月出库/领用金额
- 当前库存金额
- 当前库存品类数
- 低库存物品数量
- 负库存异常数量
- 最近操作流水
- 快捷入口：
  - 新建入库单
  - 新建出库/领用单
  - 导入 Excel
  - 导出月报
  - 创建盘点单

第一版要求：

- 首页数据必须来自数据库统计，不写死。
- 最近操作至少展示日期、业务类型、物品、数量、部门或供应商。

### 3.2 物品档案

用途：维护所有可入库、出库、统计的物品基础信息。

字段：

- 物品 ID
- 物品编码
- 物品名称
- 分类
- 规格
- 单位
- 默认单价
- 默认供应商
- 库存预警线
- 是否启用
- 备注
- 创建时间
- 更新时间

功能：

- 新增物品
- 编辑物品
- 启用/停用
- 搜索物品

- 批量导入

第一版要求：

- 支持 1000+ 物品不卡顿。
- 物品编码必须唯一。
- 名称、规格、单位、默认单价是入库和报表的核心字段。

## 5. 多电脑一致性与冗灾方案

### 5.1 总体结论

Aster 不采用“多台电脑各自保存一份数据库再同步合并”的方式。

正式多电脑使用采用局域网主机模式：

- 一台电脑作为数据主机。
- 主机保存唯一正式 SQLite 数据库。
- 主机运行本地局域网服务。
- 其他电脑作为客户端连接主机。
- 所有写入、库存计算、报表统计都由主机完成。
- 客户端不直接写数据库。

这样可以保证所有电脑看到的是同一套实时数据，避免多份 SQLite 合并带来的冲突和库存不一致。

### 5.2 运行模式

Aster 支持三种运行模式：

1. 单机模式
   - 本机保存数据库。
   - 适合只有一台电脑使用。
   - 不开放局域网连接。

2. 主机模式
   - 本机保存正式数据库。
   - 启动局域网服务。
   - 允许其他客户端连接。
   - 负责自动备份、恢复、数据健康检查。

3. 客户端模式
   - 本机不保存正式业务数据库。
   - 通过主机 IP 和端口连接主机。
   - 所有业务操作通过主机 API 完成。
   - 可保留少量本机配置，例如最近连接地址和界面设置。

第一版至少要在架构上区分这三种模式。即使 UI 先简化，也不能把数据库访问写死为“永远本机直连”。

### 5.3 一致性原则

1. 单一写入点
   只有主机可以写正式数据库。客户端提交入库、出库、盘点、调整时，必须通过主机服务。

2. 事务保护
   单据确认、库存流水生成、库存余额更新必须在主机同一个数据库事务中完成。

3. 并发控制
   多台客户端同时操作时，由主机按事务顺序处理，避免同时出库导致库存错误。

4. 版本校验
   关键编辑场景使用 updated_at 或 revision 字段做乐观锁，防止旧页面覆盖新数据。

5. 禁止共享 SQLite 文件
   不允许把 SQLite 数据库放到共享文件夹后让多台电脑直接打开写入。

### 5.4 主机服务能力

主机模式需要提供：

- 局域网 API 服务
- 连接测试接口
- 版本信息接口
- 数据库健康检查接口
- 当前连接客户端列表
- 业务 command/API
- 备份和恢复接口
- 导入导出接口

服务安全要求：

- 默认只监听局域网地址或本机配置允许的地址。
- 首次连接需要配对码或管理员确认。
- 后续客户端保存连接凭据。
- 危险操作，例如恢复备份、清空导入，必须只允许主机本机或管理员执行。

### 5.5 客户端连接能力

客户端模式需要提供：

- 主机 IP 和端口配置
- 自动发现主机，可后续实现
- 连接测试
- 断线提示
- 自动重连
- 主机不可用时禁止创建新业务单据
- 明确显示当前连接的是哪台主机

第一版可以先手动填写主机地址，后续再做局域网自动发现。

### 5.6 冗灾目标

冗灾不是只做一个“导出数据库”按钮，而是要保证出现故障时能恢复到可接受的数据状态。

目标：

- 主机电脑损坏时，可以用备份在另一台电脑恢复。
- 数据库文件损坏时，可以回滚到最近可用备份。
- 误操作发生时，可以找到操作日志并通过作废、冲正或备份恢复处理。
- 备份文件本身要可校验、可恢复、可追踪。

恢复目标：

- RPO：最多丢失当天最近一次备份之后的数据。
- RTO：普通人员按文档操作，30 分钟内恢复到另一台电脑继续使用。

### 5.7 备份策略

主机模式执行自动备份，客户端不备份正式数据库。

备份类型：

1. 启动备份
   - 每天第一次启动主机时自动备份一次。

2. 定时备份
   - 主机运行时按配置定时备份，例如每 2 小时一次。

3. 操作前备份
   - 执行 Excel 批量导入、恢复备份、数据库迁移前自动备份。

4. 手动备份
   - 管理员可随时手动创建备份。

备份内容：

- SQLite 数据库
- 应用配置
- 导入报告
- 备份元数据

备份命名：

- aster-backup-YYYYMMDD-HHMMSS.zip

备份元数据：

- 应用版本
- 数据库 schema 版本
- 创建时间
- 主机名称
- 数据库大小
- SHA256 校验值
- 备份原因：auto、manual、before_import、before_restore、before_migration

### 5.8 备份位置

建议至少保留两个位置：

1. 本机备份目录
   - 用于快速恢复。

2. 外部备份目录
   - 可以是移动硬盘、NAS、局域网共享目录或其他电脑共享目录。
   - 用于主机电脑损坏后的恢复。

第一版要求：

- 支持配置本机备份目录。
- 支持配置第二备份目录。
- 第二备份目录不可用时，系统要在首页显示告警。

### 5.9 备份保留策略

默认策略：

- 最近 7 天：保留所有自动备份。
- 最近 8 到 30 天：每天保留最后一个备份。
- 最近 12 个月：每月保留最后一个备份。
- 手动备份默认不自动删除，除非用户主动清理。

备份清理必须保留日志。

### 5.10 数据库健康检查

主机需要定期执行：

- SQLite integrity_check
- 数据库文件可读写检查
- 最近备份是否成功
- 第二备份目录是否可写
- 库存余额和流水抽样核对
- 负库存异常统计

首页需要显示：

- 最近备份时间
- 最近备份结果
- 第二备份状态
- 数据库健康状态

### 5.11 恢复流程

恢复备份必须谨慎处理。

流程：

1. 用户选择备份文件。
2. 系统校验备份包结构。
3. 系统校验 SHA256。
4. 系统显示备份创建时间、应用版本、数据库版本。
5. 系统自动备份当前数据。
6. 系统停止业务写入。
7. 系统替换数据库。
8. 系统执行数据库健康检查。
9. 系统重启或刷新主机服务。
10. 系统写入恢复日志。

恢复失败时：

- 必须回滚到恢复前自动备份。
- 必须给出失败原因。

### 5.12 主机故障切换

如果主机电脑损坏，处理方式：

1. 在另一台电脑安装 Aster。
2. 选择主机模式。
3. 从外部备份目录选择最近备份。
4. 执行恢复。
5. 新主机显示新的 IP 和端口。
6. 其他客户端修改连接地址或重新发现主机。

第一版可以手动切换主机。后续再做自动发现和一键重连。

### 5.13 不做的同步能力

第一版不做：

- 多台电脑离线分别录入后自动合并。
- 多主机写入。
- 客户端本地缓存业务单据后断网提交。
- 云端实时同步。

这些能力需要事件同步、冲突解决和权限体系，复杂度较高，不能和第一版库存正确性混在一起。

## 6. 核心业务流程

### 6.1 新增物品

1. 用户进入物品档案。
2. 点击新增物品。
3. 填写名称、分类、规格、单位、默认单价、预警线。
4. 系统校验编码唯一、必填字段完整。
5. 保存物品。
6. 物品可用于入库和出库单。

### 6.2 入库流程

1. 用户新建入库单。
2. 选择入库日期、供应商、入库类型。
3. 添加多行物品。
4. 填写数量、单价、备注。
5. 保存草稿或确认入库。
6. 确认后系统生成库存流水。
7. 系统更新库存余额和库存金额。

### 6.3 出库/领用流程

1. 用户新建出库单。
2. 选择出库日期和领用部门。
3. 添加多行物品。
4. 填写数量、用途和备注。
5. 系统检查库存是否足够。
6. 保存草稿或确认出库。
7. 确认后系统生成出库流水。
8. 系统更新库存余额。
9. 部门领用报表可立即查询。

### 6.4 月报流程

1. 用户进入报表中心。
2. 选择月份。
3. 系统按物品汇总期初、入库、出库、结存。
4. 系统按部门汇总领用数量和金额。
5. 用户查看或导出 Excel。

### 6.5 Excel 迁移流程

1. 用户选择旧 Excel 文件。
2. 系统解析文件结构。
3. 系统展示导入预览。
4. 系统列出异常和风险。
5. 用户选择导入范围。
6. 系统写入物品档案、期初和历史流水。
7. 系统生成导入报告。

## 8. 技术架构

### 8.1 前端 React

职责：

- 页面布局
- 表单交互
- 表格筛选
- 状态展示
- 调用 Tauri command
- 轻量本地状态管理

建议：

- 使用 React + TypeScript。
- 使用 Vite 作为前端构建工具。
- 表格组件要考虑 1000+ 物品和大量流水，必要时使用虚拟列表。
- 不在前端直接读写数据库。

### 8.2 Rust/Tauri 后端

职责：

- SQLite 读写
- 业务校验
- 库存计算
- Excel 导入
- Excel 导出
- 备份恢复
- 文件选择和本地路径处理
- 主机模式局域网服务
- 客户端模式远程调用主机服务

建议：

- Tauri commands 作为前后端边界。
- 核心业务逻辑放在 Rust service 层，不散落在 command 中。
- 数据库操作通过 repository 层隔离。
- 库存流水和余额更新必须在同一个事务中完成。
- 业务 service 要同时支持本机调用和主机 API 调用，避免单机模式与局域网模式写两套逻辑。

### 8.3 SQLite

职责：

- 存储业务数据
- 本地事务
- 支持备份和恢复

建议：

- 使用迁移文件管理 schema。
- 对常用筛选字段建索引：
  - item code
  - item name
  - movement date
  - department id
  - document status
  - document type

### 8.4 局域网主机服务

职责：

- 只在主机模式启动。
- 提供客户端业务 API。
- 统一处理数据库写入。
- 提供连接测试、健康检查、版本信息。
- 提供备份状态和恢复能力。

第一版建议：

- 先用 HTTP 本地服务实现，便于调试和跨客户端访问。
- 服务只暴露必要 API，不直接暴露数据库文件。
- 所有写操作都进入 Rust service 层和数据库事务。

### 8.5 备份与恢复服务

职责：

- 创建备份包
- 校验备份包
- 自动备份调度
- 备份保留清理
- 恢复前自动保护当前数据
- 恢复后健康检查

备份包格式：

- database.sqlite
- app-settings.json
- import-reports/
- backup-meta.json

### 8.6 Excel 能力

Rust 侧负责：

- 读取旧 Excel
- 解析月份 sheet
- 生成导入预览
- 导出报表 Excel

第一版重点：

- 先支持当前酒店旧表格式。
- 后续再做通用模板导入。

## 11. 关键验收标准

1. 数据正确
   任一物品在任一月份的期初、入库、出库、结存可以追溯到流水。

2. 部门领用清晰
   能快速回答“某部门某月领了哪些物品、数量多少、金额多少”。

3. 库存可信
   库存余额不是手工填的，而是由确认后的流水计算。

4. Excel 可迁移
   当前旧 Excel 至少能导入物品档案和历史流水，异常可解释。

5. 报表可交付
   月报和部门报表可以导出 Excel，并能用于内部归档。

6. 本地数据安全
   支持备份和恢复，误操作有基本保护。

7. 多电脑一致
   主机模式下，所有客户端通过同一主机写入和查询，不出现多份数据库合并问题。

8. 冗灾可恢复
   主机损坏时，可以用第二备份目录中的备份在另一台电脑恢复为新主机。
