# Aster 安全加固设计

## 目标

修复局域网客户端可伪造用户身份、密码明文保存在 WebView、默认管理员密码长期有效、自研 HTTP 服务缺少资源限制，以及密码校验阻塞数据库的问题，同时保持现有库存、报表、备份和业务数据兼容。

## 已确认的产品决策

- 采用渐进加固方案，保留现有业务 service/repository，仅替换网络与认证边界。
- 局域网业务通信使用 TLS 加密。
- 主机首次运行生成本机证书，客户端配对后固定证书身份。
- 新配对码为一次性 12 位数字，成功使用后立即轮换；配对码本身不在网络上传输。
- 设备配对与员工登录分离：设备 token 证明电脑可信，用户会话 token 证明当前操作人。
- 用户会话在主动退出或正常关闭时立即吊销；异常关闭后按闲置时限失效。
- “记住密码”使用 macOS Keychain 或 Windows Credential Manager；WebView 只保存用户名和偏好。
- 升级时删除旧 localStorage 明文密码，不迁移该密码；下次成功登录后才写入系统凭据库。
- 保留用户名、主机地址和全部业务数据；旧版设备 token 无法安全迁移，升级后每台客户端需要一次重新配对。
- 检测仍使用 `admin123` 的默认管理员，登录后强制修改密码，完成前禁止其他业务操作。

## 认证与传输架构

主机服务使用 rustls 承载现有 JSON 路由，并以有界 HTTP/1.1 解析器限制请求头、正文和响应大小。主机生成带 Aster 专用 SAN 的 Ed25519 自签名证书；私钥进入操作系统安全凭据库，无法使用凭据库时拒绝启动主机服务，不降级为明文文件。证书到期、私钥丢失或主动轮换后必须重新配对。

首次配对允许客户端为本次配对连接临时接受未知自签名证书，但配对使用 RFC 9807 OPAQUE，由 Rust `opaque-ke` 成熟实现承载，禁止自行实现密码学原语。密码套件固定为 Ristretto255 OPRF、Ristretto255 TripleDH、SHA-512 和 Argon2id KSF；协议上下文固定包含 `aster-pair-v2`、主机证书 DER 的 SHA-256、客户端设备 ID 和双方随机挑战，禁止套件协商和降级。主机生成 12 位配对码时同步生成一次性 OPAQUE 注册记录，配对码本身不在网络上传输。

配对严格执行 OPAQUE 的 KE1、KE2、KE3 三消息流程并使用 RFC 定义的双向显式认证和会话密钥。只有服务端成功验证 KE3 后，才消费当前注册记录并签发设备 token；每个交换状态超时 2 分钟释放，已消费或重放的 KE3 一律失败。设备 token 仅通过完成证书上下文绑定的 TLS/OPAQUE 通道返回，不另行设计自定义密码学封装。中间人转发到另一证书身份会因上下文和 key confirmation 不一致失败。客户端成功后固定证书指纹，后续 TLS 连接只接受该指纹。

设备配对成功后，主机签发高熵随机设备 token，仅在客户端本地设置中保存；主机数据库只保存 token 哈希。设备 token 不能表示用户身份。

员工提交用户名和密码登录后，主机签发高熵随机会话 token。主机只保存其哈希，并绑定用户、设备、创建时间、最后活动时间和过期时间。所有受保护业务路由同时验证设备 token 和会话 token，并从服务端会话记录加载当前用户与权限；删除 `X-Aster-User-Id` 信任路径。

客户端会话仅存内存，不写入磁盘。主动退出和正常关闭会尽力调用主机吊销接口；崩溃、断网和强制结束无法保证通知主机。服务端会话闲置 8 小时失效，绝对有效期 24 小时；成功授权的请求以数据库原子更新触碰最后活动时间，注销与业务请求竞态时以先提交的事务为界，注销提交后的请求全部失败。服务启动时及每 15 分钟清理过期会话。停用用户、修改密码或管理员重置密码时吊销该用户现有会话。

主机模式和单机模式继续使用进程内本地会话，但所有 Tauri command 与远程路由共享相同的授权门禁；强制改密状态在 service 层统一检查，不能通过直接调用 Tauri command 绕过。

## 默认管理员迁移

用户记录增加“必须修改密码”状态。升级迁移不会假定所有名为 admin 的账户都仍使用默认密码；启动时仅通过密码哈希验证确认当前密码确实为 `admin123`，确认后标记强制改密。新安装仍创建初始管理员，但首次登录只允许访问修改密码命令。

登录响应带回强制改密状态。前端展示不可跳过的改密界面。改密成功后清除状态、吊销旧会话，并使用新密码重新登录。除登录、退出、读取当前用户和修改本人密码之外，后端统一拒绝强制改密用户的命令，避免只依赖前端限制。

## 系统安全凭据库

新增窄接口：读取、保存和删除登录凭据。Rust 端通过跨平台系统凭据库实现，前端不直接接触存储细节。用户名仍可保存在普通本地设置；密码只经 Tauri command 进入系统凭据库。凭据按运行范围隔离：单机/主机使用本地数据库随机身份，客户端使用已固定的主机证书指纹，再组合规范化用户名作为凭据键，绝不按可变化的 IP 地址复用。

应用启动时主动删除旧的 `aster.rememberedLogin` localStorage 项。为避免再次处理旧明文，记录一次迁移标志。不会把旧明文自动写入系统凭据库。用户勾选记住密码并成功登录后才保存；取消勾选时删除，密码修改后更新。账户重命名或删除时清理可识别的旧条目；远端在其他设备重置密码无法主动删除本机条目，因此下一次认证失败时删除该范围和用户名对应的旧密码。主机地址变化不会跨主机取用密码，证书身份变化则让原条目永久失配并要求重新配对。

## 网络资源边界

- 请求头上限 16 KiB，普通 JSON 正文上限 1 MiB，单页 JSON 响应上限 8 MiB；导入导出继续走本机 Tauri 文件接口，不通过局域网传输大文件。越限返回 `413` 或 `431` 并关闭连接。
- 所有返回集合的远程路由统一分页，每页最多 500 行：库存单据和流水使用 `(业务日期, created_at, id)` 键集游标；余额、批次、盘点、审计、用户、角色、客户端、备份、审批和主数据使用与查询 scope 绑定的数据库偏移游标；报表按 11 个 section 独立分页。游标使用进程随机密钥的 HMAC-SHA256 完整性标签，并绑定规范化查询、数据库实例和对应结果集修订号；查询条件、业务数据、进程或数据库实例变化后拒绝旧游标，客户端连接心跳使用独立修订号，不误伤业务分页。所有 repository 查询都在 SQL 层使用 `LIMIT`，不先读取完整集合再内存切片。客户端 helper 自动聚合后续页，现有 React 调用仍得到原数组或 `ReportBundle`。请求正文中的单据行和盘点行最多 2,000 行；超过时前端提示拆单，不静默截断。
- TLS 连接读写超时 10 秒；远程请求的数据库执行最长 30 秒，并绑定请求取消令牌。客户端断开时令牌触发当前数据库任务的 SQLite progress handler 中断，读查询终止，写事务回滚，工作线程退出后才释放并发配额。读取超时返回可用的 `408`，数据库超时或取消映射为 `504`；若连接已断开则不再尝试发送响应。
- 全局并发连接上限 64、单 IP 上限 8；达到上限时在创建工作线程前直接关闭新连接，不再为每个连接无限创建线程。后续若切换为固定异步工作池，可在不扩大拒绝服务面的前提下于 TLS 内返回 `503`。
- 配对按 IP 限制 10 分钟 5 次失败；登录按 IP + 设备 + 规范化用户名限制 15 分钟 5 次失败；密码验证码申请和确认分别按 IP + 用户限制 15 分钟 5 次失败。越限返回 `429` 和 `Retry-After`，成功登录清除对应登录失败计数，所有失败写入安全审计。
- 对外返回稳定的 `400`、`401`、`403`、`408`、`413`、`429`、`431`、`500`、`504` 状态；`429` 包含 `Retry-After`，数据库和 IO 内部细节不返回客户端。

## 数据库并发

密码哈希计算与校验不得持有全局 SQLite 互斥锁。登录分为短查询、锁外 PBKDF2 校验、短事务写审计与创建会话三段。密码重置和改密采用相同原则，并在最终写入前重新检查用户状态，避免竞态。会话与限流逻辑使用可注入 UTC 时钟；测试时钟无等待地验证边界，生产判断允许最多 30 秒时钟回拨但绝不延长绝对 TTL。

本轮不全面替换 SQLite 单连接模型，避免扩大业务风险；只消除已确认的高成本锁内计算。

## 前端范围

本轮只拆出认证与安全凭据相关模块，避免在安全整改中同时重构全部 12,000 行界面。现有页面结构和业务交互保持不变。动画图片改为按表达组延迟加载，避免所有资源进入首屏加载路径。

## 工程化架构约束

本次整改同时建立可执行的工程规范。全部新增符号、从遗留文件迁移出的符号，以及本轮改变行为的代码路径必须满足新规范；未触及的遗留代码按可量化基线管理且不得恶化。该规则通过基线递减方式治理历史巨型文件，不以一次性重写全部存量代码为代价。

### 模块与依赖方向

前端按 `app`、`features`、`entities`、`shared` 分层：

- `app` 只负责应用装配、窗口入口、路由级状态和全局 provider。
- `features/<domain>` 包含一个业务能力的 UI、状态、用例和测试，例如 `auth`、`host-connection`、`stock`。
- `entities` 放稳定业务模型和纯领域规则，不依赖 React、Tauri 或具体传输协议。
- `shared` 只放与业务领域无关且已有至少两个真实调用方的基础能力，并按 `api`、`ui`、`lib`、`config` 细分；禁止创建无边界的 `utils.ts`、`helpers.ts` 或全局组件杂物箱。

允许的前端依赖方向为 `app -> features -> entities -> shared`。同层 feature 之间不得直接导入内部实现；跨 feature 协作通过公开入口、应用编排或领域事件。每个 feature 只通过 `index.ts` 暴露最小公共 API，内部文件不得被深层路径引用。

Rust 使用端口与适配器结构：`transport -> application -> domain`，application 同时依赖自己定义的窄 port contracts；infrastructure 实现 ports，composition root 负责装配 transport、application 与 infrastructure。domain 不依赖 application、transport 或 infrastructure。

- Tauri command 和 HTTP handler 只做反序列化、身份上下文提取、调用用例和响应映射，不包含业务 SQL。
- application service 编排权限、事务和用例，不解析 HTTP，也不操作 UI 状态；持久化、时钟、随机源、凭据库和会话存储接口放在 `application/ports`。
- domain 保持纯 Rust，承载业务值对象、校验和状态转换，不依赖 Tauri、HTTP、SQLite 或系统凭据库。
- `application/ports` 中的 repository trait 定义用例所需的最小持久化端口；SQLite、TLS、系统凭据库、生产时钟和随机源在 infrastructure 中实现这些契约。只有 composition root 可以同时导入 application 的用例和具体 infrastructure 实现进行装配。

禁止反向依赖和跨层捷径，例如 React 组件直接拼 HTTP、handler 直接写 SQL、infrastructure 反向调用 application service、domain 依赖 port/Tauri/SQLite 类型。CI 使用依赖规则脚本检查可静态识别的越层导入。

### 抽象与公共方法

- 优先保持代码靠近使用场景；只有语义相同、变化原因相同且出现至少两个真实调用方时才提取共享抽象。安全协议、时钟、随机源、凭据库和认证上下文因需要替换与测试，可在首个实现时定义窄接口。
- 公共接口必须以领域语言命名并隐藏实现细节，例如 `CredentialStore`、`SessionRepository`、`Clock`；禁止 `CommonService`、`BaseManager`、`doRequest` 等含义模糊的抽象。
- 公共函数保持单一职责，参数优先使用具名请求结构；超过 5 个独立参数时必须使用结构体/对象，避免位置参数和布尔参数组合。
- 不以复制粘贴实现协议或校验规则。序列化 DTO、领域模型和数据库记录分离，通过显式 mapper 转换；公共 mapper 不包含 I/O 或隐藏副作用。
- 禁止为了减少代码行数制造过度泛型、宏或继承层次。一个抽象如果需要调用方理解其内部特殊分支才能正确使用，应退回更具体的接口。
- 公共 API 必须有契约测试或单元测试，说明成功、失败、取消和边界行为；删除或变更公共 API 时同步更新所有调用方和文档。

### 文件、函数与状态边界

- 新建 TypeScript/Rust 源文件软上限 400 行，超过 500 行 CI 失败；测试夹具、生成文件和数据库 migration 可按清单豁免并写明原因。
- 新增函数软上限 50 行，超过 80 行 CI 失败；协议状态机或事务用例若确有必要，必须拆出具名步骤并保留复杂度测试。
- React 组件只负责展示和交互编排；I/O、缓存、协议状态和领域校验进入 hook、use case 或纯函数。禁止新增超过 15 个顶层 state 的组件。
- Rust service 不得成为跨领域入口；每个模块只对一个业务能力负责，公开符号默认保持 crate 内可见，只有稳定调用边界才使用 `pub`。
- 现有 `App.tsx`、`App.css`、`host_service.rs` 和大型 repository 建立行数基线。任何改动不得提高基线；本轮涉及的认证、凭据、TLS、会话和分页代码必须先移入独立模块，使对应基线下降。后续 CI 逐次收紧，而不是永久豁免。
- 本轮强制迁移的遗留违规只包括被替换的明文 HTTP 服务、`X-Aster-User-Id` 身份路径、登录/凭据 localStorage、默认密码门禁、锁内 PBKDF2、无界连接线程和新增分页路径。其他未改变行为的旧 UI、业务 route 与 repository 实现只要求行数、lint 和依赖违规基线不增加；后续任务再分批迁移。

### 错误、并发与副作用

- TypeScript 不允许新增 `any`、浮动 Promise、空 catch 或以字符串猜测错误类型；应用错误使用可判别 union，并在边界统一映射为用户消息。
- Rust 生产代码禁止新增 `unwrap`、`expect`、`panic!` 和静默丢弃错误；锁中毒、后台任务退出、取消和超时必须转换为结构化错误并记录上下文。
- 所有数据库多步写入使用显式事务；事务边界在 application/repository 用例中可见，不隐藏在通用 helper 中。外部网络、PBKDF2、文件和凭据库 I/O 不得在持有 SQLite 全局锁时执行。
- 后台任务必须有所有者、取消令牌和关闭路径；禁止 detached 无限线程。日志不得包含密码、配对码、设备 token、会话 token、验证码或完整凭据键。

### 测试与质量门禁

- 新增领域规则先写纯单元测试；repository 使用临时 SQLite 集成测试；HTTP/TLS/OPAQUE 使用协议级测试；前端认证状态与凭据迁移使用组件或 hook 测试。
- 修复缺陷必须包含能在修复前失败的回归测试。安全拒绝路径与成功路径同等要求覆盖。
- 测试使用可注入时钟、随机源、凭据库和网络边界，不依赖真实等待、真实系统凭据或公网。
- CI 必须执行 TypeScript 类型检查、ESLint、前端测试、Vite 构建、`cargo fmt --check`、严格 Clippy、Rust 测试、依赖审计、迁移测试、依赖方向检查和文件基线检查。
- 禁止通过全局 `allow`、降低 lint 等级、扩大豁免清单或删除测试绕过门禁；局部豁免必须附带具体理由、负责人语义和清理条件。

### 完成定义

工程化整改完成不仅意味着功能可运行，还必须满足：新增安全模块符合依赖方向；公共接口有测试；历史巨型文件基线下降；不存在新增重复协议实现；严格 lint 与全部测试通过；架构检查可在本地和 CI 重复执行；工程规范文档成为仓库长期维护入口。

## 兼容与失败处理

- 数据库迁移保持向前兼容，升级前沿用现有自动备份策略。
- 新版本不再启动旧明文 HTTP 端点，也不接受旧设备 token 或 `X-Aster-User-Id`。新客户端保留主机地址和设备名称，但清除旧 token，并明确提示一次性重新配对。旧客户端连接新主机只会因 TLS 握手失败而显示其原有连接失败信息；升级说明和主机界面明确要求先升级所有客户端。不会为旧客户端保留任何明文提示端口。
- TLS 私钥丢失、证书缺失或证书与私钥不匹配时，主机拒绝启动安全服务；管理员显式重置主机身份后才生成新证书，客户端明确提示主机身份变化并要求重新配对，绝不静默接受。
- 系统凭据库不可用时登录仍可继续，但“记住密码”失败并显示可操作提示，不回退到明文存储。
- 会话过期统一回到登录页，不丢失已在本机编辑但尚未提交的数据。

## 验证策略

- 单元测试：会话签发、仅存哈希、设备绑定、闲置 8 小时、绝对 24 小时、30 秒回拨容差、吊销竞态、默认密码识别、强制改密门禁、各限流窗口和所有请求上限。
- 协议测试：OPAQUE 错误上下文、证书哈希变化和 KE3 重放均失败且不签发 token；伪造或提供 `X-Aster-User-Id` 不再影响身份；缺少任一 token 返回 `401`；权限不足返回 `403`；证书变化被拒绝；配对码只能使用一次；旧 HTTP、旧 token 和旧协议无法访问业务数据。
