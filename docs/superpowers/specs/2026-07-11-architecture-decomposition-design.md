# Aster 大文件拆分设计

## 目标

在不改变界面、业务规则、数据格式和远程协议的前提下，消除单文件职责集中。拆分完成后，新模块保持单一业务职责，现有大文件行数基线只允许下降，并为后续逐步降到 500 行以内建立稳定边界。

## 方案选择

不采用只移动代码但继续共享巨型上下文的“文件切片”，也不在本轮重写状态管理或数据库模型。采用按业务域纵向拆分：先抽取稳定类型、纯函数、共享 UI 和独立页面，再拆后端远程客户端、HTTP 路由、认证与库存事务模块。这样能保持行为不变，同时让模块拥有明确输入输出。

## 前端结构

- `app`：应用入口、全局会话、导航和窗口编排。
- `features/<domain>`：仪表盘、库存、盘点、主数据、报表、导入、设置、备份、日志和编辑器。
- `entities`：跨功能共享的领域 DTO 与展示模型。
- `shared`：格式化、搜索、权限、外观、基础表格和表单控件。
- 样式按基础、布局和业务功能拆分，由入口统一导入，组件不依赖巨型全局样式文件。

## Rust 结构

- `host_service` 仅保留主机生命周期编排；远程客户端、路由分派、认证/会话和客户端连接持久化分别进入子模块。
- `stock_repository` 按单据、查询、FIFO 批次、预算校验和撤销拆分；事务边界保留在用例入口。
- `import_service` 按解析、预览、执行、模板和报告拆分。
- `backup_service` 按创建、恢复、归档验证和保留策略拆分。

## 约束

- 不改变 Tauri command 名称、JSON 字段、SQLite schema 语义和 UI 文案。
- 跨模块只导出真实调用方需要的最小接口；禁止新增万能 utils 或公共上下文对象。
- 新文件不超过 500 行；拆出的模块必须通过 TypeScript、Clippy 和现有测试。
- 每完成一个领域后运行针对性检查，最终运行生产构建、工程门禁、依赖审计和全部 Rust 测试。

## 验收

- `App.tsx`、`App.css`、`host_service.rs`、`stock_repository.rs`、`import_service.rs` 和 `backup_service.rs` 均显著下降。
- 工程基线同步收紧且不得提高其他文件上限。
- 所有现有功能测试通过；生产构建和严格 Clippy 通过；工作区无生成物或临时脚本。
