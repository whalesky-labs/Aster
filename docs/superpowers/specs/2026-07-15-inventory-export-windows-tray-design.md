# Aster 供应商筛选、库存全量导出与 Windows 托盘设计

日期：2026-07-15

## 目标

一次性完成三个相互独立但共同影响桌面端主流程的需求：物品档案支持按供应商筛选；库存台账支持最高权限管理员导出全量库存核对表；Windows 主窗口支持系统托盘。实现必须覆盖独立运行、主机和客户端三种运行模式，保持现有列表布局、权限体系、远程协议安全边界和工程化约束。

## 已确认选择

采用完整工程化方案，不采用仅前端筛选、导出当前页或复用月度报表的简化方案。

- 现有 `admin` 角色是系统最高权限，本需求中的“超级管理员”对应 `role.code === "admin"`，不新增第二套角色或迁移现有账号。
- 供应商筛选必须进入 SQLite 和主机 API 的真实分页查询，不能只过滤当前已加载行。
- “导出全部库存”忽略当前库存页面的分类、物品、库存状态筛选以及分页游标，导出数据库中的全部物品库存快照。
- Windows 关闭主窗口时隐藏到托盘；只有托盘菜单中的“退出”才结束应用。编辑子窗口仍按原行为关闭。

## 范围与非目标

### 包含

- 物品档案供应商下拉筛选、清空、分页重置和筛选后导出一致性。
- 独立的库存全量 Excel 导出用例、管理员双重鉴权、主机审计和客户端本地落盘。
- Windows 主窗口托盘图标、显示主界面、点击恢复和明确退出。
- 前端、Rust 服务、SQLite 查询、远程路由和 Windows 编译测试。

### 不包含

- 不新增“超级管理员”角色或权限配置页面。
- 不改变普通用户查看库存台账的现有权限。
- 不导出逐笔库存流水、批次明细或月度统计；这些继续使用现有页面和报表。
- 不改变 macOS 的窗口关闭行为，也不把编辑子窗口隐藏到托盘。
- 不引入通用任务队列、后台导出中心或新的 Excel 库。

## 1. 物品档案供应商筛选

### 界面与状态

`ItemsPage` 在现有关键字搜索区域增加标准 `<select>` 下拉菜单，标签为“供应商”，首项为“全部供应商”，选项使用已经加载的启用供应商。关键字和供应商组成同一个物品查询状态；点击搜索提交两项条件，点击清空恢复空关键字和全部供应商。

供应商改变后不直接对 `items` 数组做前端过滤，而是提交新查询、清空旧数据并重置分页游标。`PaginatedTable.resetKey` 同时包含关键字和供应商 ID，避免筛选变化后保留旧的滚动分页状态。

物品档案工具栏中的现有“导出”沿用当前筛选条件，同时传递关键字和供应商 ID，保证屏幕列表与导出结果语义一致。

### 数据接口

物品列表和物品导出接口增加可选 `supplierId`：

- Tauri commands：`list_items`、`list_items_page`、`export_items`。
- 服务层：本地查询和客户端远程查询使用同一组筛选条件。
- 主机 API：`GET /api/master/items` 读取可选 `supplierId`。
- SQLite：`master_items.supplier_id` 使用参数化等值条件；分页 scope 必须包含供应商 ID，防止不同筛选共用游标。

旧调用不传 `supplierId` 时行为保持不变。

### 异常处理

- 下拉选项列表为空时只显示“全部供应商”，物品查询仍可正常使用。
- 已停用供应商不作为新筛选选项显示；若状态中存在已失效 ID，提交后后端按该 ID 查询，不产生崩溃或越权。
- 查询或导出失败沿用主界面统一错误区，不静默吞错。

## 2. 库存台账全量导出

### 权限边界

库存台账顶部增加“导出全部库存”按钮，仅 `admin` 用户可见。权限不能依赖按钮隐藏：

- 前端只负责按钮可见性；Tauri command 不自行复制权限规则，而是立即委托 service。
- Service 是本地权限的唯一执行点，在任何目录操作或数据读取前调用一次 `require_admin`。
- 客户端模式使用专用主机 API；主机 API 调用 `require_remote_admin`，拒绝普通远程账号直接请求。

普通用户手工调用 command 或远程端点时返回明确的权限错误，不生成文件，也不写成功审计。

### 导出数据模型

新增只服务于导出的 `StockBalanceExportRow`，不把展示页面模型扩成混合职责。SQLite 使用单次一致性读取生成全量快照，基于 `master_items` 左连接库存余额和基础资料，因此包含：

- 启用和停用物品；
- 正库存、零库存和负库存；
- 尚无 `stock_balances` 行的物品，其数量和金额按零处理。

Excel 工作表名为“库存台账”，文件名为 `Aster-库存台账-YYYYMMDDHHMMSS.xlsx`。列顺序固定为：物品编码、物品名称、分类、规格、单位、供应商、库存数量、平均成本、库存金额、最后入库价、预警数量、库存状态、物品状态。金额和数量使用数值单元格及统一格式，不写成字符串；首行加粗并设置可读列宽。

“全量”明确忽略页面当前筛选和分页，不接受查询参数，避免管理员误以为拿到完整数据但实际只导出了当前视图。

### 本地与客户端数据流

独立/主机模式：service 校验管理员身份，在本地数据库读取全量导出行，写入系统有效导出目录，成功后写一条 `export_stock_balances` 审计日志，摘要包含导出行数。

客户端模式：service 校验当前登录用户为管理员后，首先从客户端本地数据库解析 `default_export_dir`；未配置时回退到当前客户端 `AppPaths.export_dir`。目录创建和可写性检查必须在请求远程导出之前完成，且绝不使用主机系统设置或主机返回的目录。

客户端随后调用专用管理员端点 `GET /api/stock/balances/export`。主机在一次 SQLite 读快照中取得全部导出行，使用同一 workbook writer 生成 XLSX 内存缓冲区，并以二进制响应返回；该端点不接收筛选、分页或游标参数。客户端将完整响应原子写入已经验证的本地目录并返回当前电脑可访问的路径，避免 Windows 客户端拿到不可访问的主机路径，也避免跨页查询期间库存变化导致导出内容不一致。

库存台账导出是明确的非分页批处理：无行数上限，不复用界面游标接口。单次 SQLite SELECT 在语句生命周期内提供一致视图；工作簿在完整生成后才响应或落盘。内存分配、工作簿生成或传输失败均返回错误且不产生部分 Excel。该选择以完整一致性为优先，符合管理员人工核库的低频场景。

远程审计分清“数据访问”和“文件生成”：主机在管理员鉴权通过、快照及工作簿成功生成后记录 `read_stock_balance_export`，表示该远程管理员已获准读取库存导出数据；operator 记录实际登录管理员，摘要同时包含配对客户端标识和导出行数。客户端本地文件写入失败不会删除这条敏感数据访问记录，也不把它描述成文件生成成功。本地独立/主机模式的 `export_stock_balances` 只在文件成功保存后记录。远程模式不增加成功回执端点，避免为低频导出引入分布式事务。

导出职责分为三个独立单元：

1. Repository 只负责生成完整、确定排序的库存快照。
2. Workbook writer 只负责把快照写成文件或内存 XLSX 缓冲区。
3. Service 只负责编排权限、运行模式、保存目录和审计。

### 一致性与失败策略

- 数据读取失败时不创建文件。
- 本地 Excel 写入失败返回具体错误，不写 `export_stock_balances` 成功审计，并删除临时文件；远程主机的数据访问审计按上一节的独立语义保留。
- 客户端网络失败时不产生空表或部分表。
- 文件名使用 Windows 安全字符。
- 库存状态严格沿用现有判定：数量小于 0 为负库存；数量大于等于 0 且小于等于预警数量为低库存；其余为正常。
- 导出按物品启用状态降序、物品编码升序、物品 ID 升序确定排序，重复执行结果顺序稳定。

## 3. Windows 系统托盘

### 平台行为

托盘代码使用 `cfg(target_os = "windows")` 隔离，并启用 Tauri 2 所需的 tray feature。macOS 和其他平台不注册托盘，也不修改现有关闭行为。

Windows 启动时创建单个托盘图标，复用应用图标。托盘菜单包含：

- “显示 Aster”：显示主窗口、取消最小化并聚焦。
- “退出”：调用应用退出，真正结束进程。

托盘图标左键释放时执行“显示 Aster”。不依赖双击，保证单击即可恢复；重复操作保持幂等。

### 窗口事件

仅拦截 label 为 `main` 的窗口关闭请求：阻止默认关闭并隐藏主窗口。编辑窗口和将来新增的非主窗口不拦截，继续正常销毁。

托盘“退出”唯一使用 Tauri 应用级正常退出 API `AppHandle::exit(0)`，不调用主窗口 `close()`，也不引入共享退出标记。主窗口关闭拦截只响应 `WindowEvent::CloseRequested`，因此不会把应用级退出转换成隐藏。若主窗口尚未创建完成或恢复聚焦失败，记录可诊断错误但不让托盘事件线程 panic。

### 生命周期约束

- 不创建重复托盘实例。
- 隐藏主窗口不清理登录状态、主机服务或定时备份任务。
- 托盘退出沿用 Tauri 正常退出流程，不强制终止线程或绕过资源清理。

## 4. 工程结构

新增逻辑按领域放置，避免继续扩大主文件：

- 物品筛选参数和分页查询保持在 master-data 领域。
- 库存导出模型放在 stock domain，查询放在 stock repository，Excel writer 放在 stock service 子模块。
- 托盘构建和主窗口事件放在独立 desktop/tray 模块，`lib.rs` 仅负责装配。
- React 页面只接收业务 props，不直接感知远程 HTTP 或文件系统。

所有生产源码继续满足单文件不超过 500 行；新增公共函数应有单一职责，避免复制现有 Excel 错误映射、窗口恢复或查询参数编码逻辑。

## 5. 测试与验收

### 自动化测试

- Repository：供应商条件只返回对应物品；分页 scope 在供应商变化后隔离；无供应商条件保持旧行为。
- UI/状态：供应商下拉提交正确 ID，清空恢复全部，分页 reset key 包含两项条件。
- Repository：库存导出包含零库存、负库存和停用物品，字段及排序正确。
- Service：普通用户导出被拒绝且不产生文件；管理员导出生成有效 `.xlsx`，列标题、行数、数值单元格和审计日志正确。
- Remote API：普通远程用户被拒绝，远程管理员可取得全量快照并写主机审计。
- Tray：平台无关的菜单事件/窗口判定逻辑做纯函数测试；Windows CI 编译实际托盘装配代码。

### 完整验证

- 前端 TypeScript build、Vitest、Playwright 和工程规范检查。
- Rust fmt、Clippy、全部 Rust 测试。
- macOS 本地 Tauri 构建确认条件编译没有影响当前客户端。
- GitHub Actions Windows target 执行 `cargo check`，确认 tray feature、图标和窗口 API 可编译。
- 最终人工 review：三个入口均可发现，列表布局仍符合“页面特有功能 → 搜索 → 表格”，非管理员看不到导出按钮，编辑子窗口关闭行为不受托盘影响。
- Windows 人工验收：主窗口关闭后进程仍在且可由托盘恢复；托盘退出后进程确实结束；应用更新、操作系统会话退出等应用级退出流程不被主窗口关闭拦截阻止。

## 完成标准

1. 物品档案可通过标准下拉菜单按供应商查询完整分页结果，清空和导出语义一致。
2. 只有 `admin` 能生成包含全部物品库存状态的 Excel，独立、主机和客户端模式均返回当前电脑可访问的文件路径，并有可追溯审计。
3. Windows 主窗口关闭后应用留在托盘，单击恢复，选择退出后进程结束；子窗口和 macOS 行为不回归。
4. 新增测试与现有全量验证通过，生产文件保持不超过 500 行。
